# M1 Foundation — Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Bootstrap the Better Hayabusa/ChainSaw Tauri app so it launches on Windows, blocks gracefully on non-Windows, renders the sidebar shell with the Merciless Software footer, opens the About page showing the version from `tauri.conf.json`, and has a working `app.db` (next to the exe) with a migration runner.

**Architecture:** Tauri 2 desktop app, Rust backend owning all heavy work, SvelteKit + shadcn-svelte frontend that only renders. All cross-language data flows through `#[tauri::command]` handlers with `ts-rs`-generated TS types. SQLite via `rusqlite` (no `plugin-sql`). OS-gate component at the layout root short-circuits the app on non-Windows.

**Tech Stack:** Tauri 2.x, Rust (rusqlite + tokio + serde + ts-rs + tracing), SvelteKit (`adapter-static`), shadcn-svelte, Tailwind CSS 4, pnpm, Vitest, Playwright (deferred to later milestones).

**Spec reference:** [docs/superpowers/specs/2026-05-19-better-hayabusa-chainsaw-design.md](../specs/2026-05-19-better-hayabusa-chainsaw-design.md) — see sections 2, 3, 4a, 5, 5a, 8a.

### Task 4: App-data path resolution (next-to-exe in release, dev fallback)

**Files:**
- Create: `src-tauri/src/paths.rs`
- Modify: `src-tauri/src/lib.rs`
- Create: `src-tauri/tests/paths.rs`

The portable model puts `app.db` next to the executable. In `cargo run` / `tauri dev` the "executable" is in `src-tauri/target/debug/`, which is fine for dev — but we want a sane dev fallback that doesn't pollute `target/`. Resolution:

- **Release**: `<exe_parent>/app.db` (true portable)
- **Dev**: `<exe_parent>/app.db` still — `target/debug/app.db` is acceptable; it's git-ignored and removable

We'll provide a single resolver function and unit-test it with a tempdir.

- [ ] **Step 3: Implement `AppPaths`**

Create `src-tauri/src/paths.rs`:

```rust
use std::path::{Path, PathBuf};

use thiserror::Error;

#[derive(Debug, Error)]
pub enum PathError {
    #[error("executable path has no parent directory: {0}")]
    NoParent(PathBuf),
}

#[derive(Debug, Clone)]
pub struct AppPaths {
    pub exe_dir: PathBuf,
    pub app_db: PathBuf,
    pub tools_dir: PathBuf,
}

impl AppPaths {
    pub fn from_exe(exe: &Path) -> Result<Self, PathError> {
        let exe_dir = exe
            .parent()
            .ok_or_else(|| PathError::NoParent(exe.to_path_buf()))?
            .to_path_buf();

        Ok(Self {
            app_db: exe_dir.join("app.db"),
            tools_dir: exe_dir.join("tools"),
            exe_dir,
        })
    }

    pub fn from_current_exe() -> Result<Self, PathError> {
        let exe = std::env::current_exe()
            .map_err(|_| PathError::NoParent(PathBuf::from("<current_exe failed>")))?;
        Self::from_exe(&exe)
    }
}
```

The migration files live in the source tree and need to be reachable from the compiled binary. Use Rust's `include_str!` so the SQL ships in the binary.

Create `src-tauri/src/db/migrations.rs`:

```rust
use rusqlite::{Connection, OptionalExtension};
use thiserror::Error;

#[derive(Debug, Error)]
pub enum MigrationError {
    #[error("sqlite error: {0}")]
    Sql(#[from] rusqlite::Error),
    #[error("migration {0} failed: {1}")]
    Migration(&'static str, rusqlite::Error),
}

/// A single migration: a stable name + its SQL.
pub struct Migration {
    pub name: &'static str,
    pub sql: &'static str,
}

pub const APP_MIGRATIONS: &[Migration] = &[Migration {
    name: "001_init",
    sql: include_str!("../../migrations/app/001_init.sql"),
}];

pub fn run_migrations(conn: &Connection, migrations: &[Migration]) -> Result<(), MigrationError> {
    // Ensure the bookkeeping table exists before checking which migrations have run.
    conn.execute(
        "CREATE TABLE IF NOT EXISTS _migrations (name TEXT PRIMARY KEY, applied_at TEXT NOT NULL)",
        [],
    )?;

    for m in migrations {
        let already_applied: Option<String> = conn
            .query_row(
                "SELECT name FROM _migrations WHERE name = ?1",
                [m.name],
                |row| row.get(0),
            )
            .optional()?;

        if already_applied.is_some() {
            continue;
        }

        let tx = conn.unchecked_transaction()?;
        tx.execute_batch(m.sql).map_err(|e| MigrationError::Migration(m.name, e))?;
        tx.execute(
            "INSERT INTO _migrations (name, applied_at) VALUES (?1, datetime('now'))",
            [m.name],
        )?;
        tx.commit()?;
    }

    Ok(())
}
```

```rust
use std::path::Path;

use rusqlite::Connection;

use crate::db::migrations::{run_migrations, MigrationError, APP_MIGRATIONS};

#[derive(Debug, thiserror::Error)]
pub enum AppDbError {
    #[error("sqlite: {0}")]
    Sql(#[from] rusqlite::Error),
    #[error("migration: {0}")]
    Migration(#[from] MigrationError),
}

/// Opens (creating if missing) the app.db at the given path and runs migrations.
pub fn open_or_create(db_path: &Path) -> Result<Connection, AppDbError> {
    let conn = Connection::open(db_path)?;
    conn.pragma_update(None, "foreign_keys", "ON")?;
    conn.pragma_update(None, "journal_mode", "WAL")?;
    run_migrations(&conn, APP_MIGRATIONS)?;
    Ok(conn)
}
```

```typescript
import { platform as tauriPlatform } from '@tauri-apps/plugin-os';

export type SupportedPlatform = 'windows' | 'macos' | 'linux' | 'other';

let detected: SupportedPlatform | null = null;

export async function detectPlatform(): Promise<SupportedPlatform> {
  if (detected !== null) return detected;
  const raw = await tauriPlatform();
  detected = raw === 'windows' || raw === 'macos' || raw === 'linux' ? raw : 'other';
  return detected;
}

export function getCachedPlatform(): SupportedPlatform | null {
  return detected;
}
```

### Risks / things to watch during execution

- **`pnpm create tauri-app` into a non-empty directory** may refuse. If so, Task 1 Step 1 includes the fallback (scaffold in temp dir, move into root preserving spec/CLAUDE.md/scripts). Tested mentally; not yet exercised.
- **`ts-rs` codegen** depends on tests running. If a developer never runs `cargo test`, the frontend will be missing types and TypeScript will error. Build pipelines in later milestones should call `cargo test --test ts_export` explicitly.
- **`rusqlite` with `bundled`** compiles SQLite from source on first build (~30s). Expect a slow first `cargo build`. Subsequent builds are incremental.
- **`pnpm tauri dev` first launch** can take a couple of minutes (Rust compile + Vite warmup). This is normal.
